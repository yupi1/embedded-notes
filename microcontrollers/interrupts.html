Переривання

Головна ідея: переривання замінює polling. Замість того щоб постійно питати "є дані? є дані?" — CPU займається справами і отримує сигнал лише коли подія сталась. Аналогія: дзвінок телефону замість того щоб щохвилини перевіряти чи хтось подзвонив.
02 — Принцип роботи

Кожне переривання проходить 7 чітких кроків від тригера до повернення у програму.

  • Подія або сигнал (тригер) виникає всередині MCU або ззовні.
  • Встановлюється відповідний прапорець переривання у регістрі стану.
  • CPU завершує поточну інструкцію і призупиняє виконання коду.
  • CPU зберігає контекст на стек — адресу повернення (PC) і регістри загального призначення.
  • CPU читає адресу ISR з векторної таблиці і переходить до неї.
  • Після завершення ISR контекст відновлюється зі стека.
  • Програма продовжує виконання рівно з того місця, де зупинилась.

04 — Переваги
  • Ефективність — CPU не витрачає час на постійний polling. Займається корисною роботою між подіями.
  • Швидкий відгук — реакція відбувається апаратно, без затримки main loop. Критично для real-time систем.
  • Асинхронність — кілька ISR можуть обробляти різні події незалежно: UART, таймер, GPIO — у кожного своя ISR.
  • Енергозбереження — MCU може входити в режим сну (WFI — Wait For Interrupt) і прокидатись тільки за перериванням. Основа low-power систем.

06 — Правила використання та обмеження
Пріоритети — NVIC (ARM Cortex-M)

На ARM Cortex-M управління перериваннями реалізовано через NVIC (Nested Vectored Interrupt Controller). Слово "Nested" ключове — переривання з вищим пріоритетом може перервати ISR з нижчим. Пріоритет 0 = найвищий, 15 = найнижчий.

ПріоритетРівеньТипові призначення
0–2КритичнийАварійна зупинка, безпека, watchdog, hard realtime
3–7ВисокийUART, SPI, I²C, таймери управління моторами
8–15НизькийКнопки, індикатори, логування, некритичні події
Правила написання ISR
Правило 01
ISR має бути максимально короткою
Поки виконується ISR, інші переривання з нижчим або рівним пріоритетом заблоковані. Довга ISR = затримки = втрата сигналів. Правильний патерн: ISR встановлює volatile прапорець або пише в чергу — вся логіка виконується в main loop або FreeRTOS task.
Правило 02
Заборонено: delay(), malloc(), printf()
HAL_Delay() чекає SysTick — але SysTick може мати нижчий пріоритет ніж поточна ISR → deadlock.
malloc() — небезпечний у ISR через не-reentrant heap.
printf() — використовує глобальний стан, не reentrant. На ESP32 замість xQueueSend — xQueueSendFromISR.
Правило 03
Всі спільні змінні — volatile
Компілятор може кешувати значення звичайної змінної в регістрі CPU. Якщо ISR змінює цю змінну — main ніколи не побачить нового значення. volatile забороняє кешування і гарантує читання актуального значення з RAM.
Правило 04
Атомарність багатобайтових змінних
На 8-бітному MCU читання uint16_t або uint32_t займає кілька інструкцій. Якщо ISR змінює змінну між цими інструкціями — дані будуть пошкоджені. Захист: вимкнення переривань на час читання або атомарні операції.
Правило 05
Не викликати non-reentrant функції
Non-reentrant — функція, яка зберігає стан у глобальних або статичних змінних (наприклад, strtok()). Якщо main викликає таку функцію і ISR перериває її всередині — обидва виклики паразитують на одному стані → непередбачувана поведінка.
WRN
Золоте правило: ISR встановлює прапорець — main обробляє. Якщо ти бачиш складну логіку в ISR — це знак поганої архітектури.
07 — Код
C · STM32 HAL — зовнішнє переривання GPIO + правильний патерн 
// 1. Налаштування GPIO на EXTI (зовнішнє переривання по фронту)
GPIO_InitTypeDef gpio = {0};
gpio.Pin  = GPIO_PIN_0;
gpio.Mode = GPIO_MODE_IT_RISING;   // переривання по наростаючому фронту
gpio.Pull = GPIO_PULLDOWN;
HAL_GPIO_Init(GPIOA, &gpio);

// 2. NVIC — задати пріоритет і увімкнути
HAL_NVIC_SetPriority(EXTI0_IRQn, 5, 0);  // preempt=5, sub=0
HAL_NVIC_EnableIRQ(EXTI0_IRQn);

// 3. Прапорець — volatile обов'язково!
volatile uint8_t button_flag = 0;

// 4. HAL Callback — викликається автоматично з EXTI IRQ Handler
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
    if (GPIO_Pin == GPIO_PIN_0) {
        button_flag = 1;  // тільки прапорець — ніяких delay(), printf() тут!
    }
}

// 5. Обробка в main loop — правильна архітектура
while (1) {
    if (button_flag) {
        button_flag = 0;
        handle_button();   // вся логіка тут, а не в ISR
    }
}
C · ESP32 ESP-IDF — GPIO interrupt + FreeRTOS Queue 
// IRAM_ATTR — функція в IRAM (швидка пам'ять), доступна навіть при Flash кешуванні
static void IRAM_ATTR gpio_isr_handler(void* arg) {
    uint32_t gpio_num = (uint32_t) arg;
    // xQueueSendFromISR — ISR-безпечна версія (не блокує!)
    xQueueSendFromISR(gpio_queue, &gpio_num, NULL);
}

// Реєстрація ISR
gpio_install_isr_service(0);
gpio_isr_handler_add(GPIO_NUM_0, gpio_isr_handler, (void*)GPIO_NUM_0);

// FreeRTOS task — обробляє подію поза ISR
void gpio_task(void* arg) {
    uint32_t io_num;
    for (;;) {
        if (xQueueReceive(gpio_queue, &io_num, portMAX_DELAY)) {
            handle_gpio(io_num);   // безпечна обробка без обмежень
        }
    }
}
08 — Питання на співбесіді

Polling — CPU постійно перевіряє умову: while(!ready);. Простий, але CPU повністю заблокований. Є затримка реакції — рівна часу одного оберту циклу.

Interrupt — CPU займається корисною роботою, апаратне переривання викликає ISR в момент події. Реакція майже миттєва (кілька тактів на збереження контексту). CPU вільний між подіями.

Polling виправданий лише для дуже коротких і передбачуваних очікувань (наприклад, чекати 1–2 такти CPU на флаг готовності SPI). В усіх інших випадках переривання ефективніші.

Векторна таблиця — масив 32-бітних адрес ISR у Flash, починаючи з адреси 0x08000000 на STM32 (або 0x00000000 після ремапінгу). Кожна позиція відповідає конкретному типу переривання.

Коли виникає переривання, CPU автоматично читає адресу ISR з відповідної позиції таблиці і переходить туди. Це апаратно, без участі програміста — тому і "vectored". На STM32 таблиця генерується у startup_stm32xxx.s.

Компілятор оптимізує код. Якщо всередині циклу значення змінної явно не змінюється — компілятор може "підняти" читання перед циклом і кешувати в регістрі. Цикл потім читає старе значення з регістра, ніколи не бачить оновлення з ISR.

volatile забороняє таку оптимізацію — кожне звернення іде напряму в RAM. Це єдиний надійний спосіб передавати дані між ISR і main без атомарних примітивів.

NVIC (Nested Vectored Interrupt Controller) — апаратний модуль ARM Cortex-M. Приймає запити від усіх джерел, визначає пріоритет, дозволяє вкладення (нижчий номер = вищий пріоритет).

Налаштування через HAL:

HAL_NVIC_SetPriority(EXTI0_IRQn, 5, 0); — preempt пріоритет 5, sub-пріоритет 0.

HAL_NVIC_EnableIRQ(EXTI0_IRQn); — увімкнути.

PreemptPriority визначає хто може перервати кого (вкладення). SubPriority — порядок обробки при рівному preempt (не дозволяє вкладення).

HAL_Delay() всередині чекає на SysTick_Handler — системне переривання з певним пріоритетом. Але якщо поточна ISR має пріоритет рівний або вищий за SysTick — SysTick ніколи не спрацює, лічильник не збільшиться, HAL_Delay буде чекати вічно. Deadlock.

Те ж саме стосується будь-яких блокуючих функцій: vTaskDelay(), HAL_UART_Transmit() без DMA, і т.д.

NVIC обробляє ситуацію детерміновано. Порівнюється preempt-пріоритет: менший номер = вищий пріоритет — обробляється першим. Друге чекає.

Якщо preempt-пріоритет однаковий — порівнюється subpriority. При повній рівності — менший номер вектора має вищий апаратний пріоритет.

Важливо: при рівному preempt-пріоритеті одне переривання не може перервати інше, навіть якщо воно вже запущено. Тільки менший preempt дозволяє вкладення.
