Universal Asynchronous
Receiver-Transmitter

Асинхронний послідовний інтерфейс для двосторонньої комунікації між двома пристроями. Не потребує тактового сигналу — обидві сторони домовляються про швидкість заздалегідь.

01 — Визначення

UART (Universal Asynchronous Receiver-Transmitter) — це апаратний модуль мікроконтролера, який реалізує асинхронний послідовний інтерфейс для обміну даними між двома пристроями. "Асинхронний" означає, що немає окремого тактового сигналу — на відміну від SPI або I²C. Замість цього обидві сторони домовляються про baud rate заздалегідь і кожна незалежно відраховує час між бітами.

KEY
Головна ідея: UART — це домовленість про темп. Передавач і приймач говорять з однаковою швидкістю, але кожен окремо. Стартовий біт — це сигнал "починаємо!", після якого кожен сам рахує такти.
02 — Лінії зв'язку

Базовий UART використовує лише дві лінії. Для управління потоком (Hardware Flow Control) додаються ще дві.

TX
Transmit обов'язкова
Пін передачі. Дані виходять звідси і йдуть до RX іншого пристрою. TX першого → RX другого.
RX
Receive обов'язкова
Пін прийому. Приймає дані з TX іншого пристрою. Підключається до TX партнера.
RTS
Request To Send flow control
Сигналізує партнеру: "я готовий приймати дані". Коли піднятий HIGH — прийом відкритий.
CTS
Clear To Send flow control
Вхідний сигнал дозволу. Якщо партнер тягне LOW — передача зупиняється (він переповнений).
WRN
Типова помилка: TX підключається тільки до RX, а не TX до TX. Якщо обидва TX'и з'єднані між собою — шина "коротить" і дані губляться.
03 — Налаштування

Порядок конфігурації UART незалежно від платформи завжди однаковий.

  • Встановити baud rate — швидкість передачі в бодах. Типові значення: 9600, 38400, 115200, 230400.
  • Налаштувати Data Frame — кількість біт даних (зазвичай 8), парність (none/even/odd), стоп-біти (1 або 2).
  • Увімкнути TX та/або RX — можна ввімкнути тільки передачу або тільки прийом.
  • Опційно: увімкнути переривання — щоб CPU не чекав а отримував колбек при появі даних.
Розрахунок UBRR (AVR)

Оскільки UART асинхронний, він не використовує окремий тактовий сигнал. Замість цього кожен пристрій сам відраховує тривалість одного біта. Кількість тактів CPU на один бод обчислюється за формулою:

Формула — UBRR (AVR, async normal mode) 
// Prescaler = 16 (нормальний режим, U2X=0)
UBRR = (F_CPU / (16 × BAUD)) - 1

// Приклад: F_CPU = 16 МГц, BAUD = 9600
UBRR = (16_000_000 / (16 × 9600)) - 1 = 103

// Тобто: UART відраховує рівно 104 такти CPU на кожен бод
// "−1" тому що лічильник рахує від 0, а не від 1 → 0..103 = 104 значення

Де F_CPU — тактова частота мікроконтролера, BAUD — бажана швидкість, 16 — prescaler (ділник), константа для нормального асинхронного режиму. У подвійно-швидкому режимі (U2X=1) prescaler дорівнює 8 — це дозволяє отримати точніший baud rate на деяких частотах.

WRN
Якщо частоти передавача і приймача відрізняються більше ніж на 2–3%, UART починає передавати помилкові дані — приймач не може точно визначити межі бітів. Тому для UART завжди краще кварцовий генератор або PLL, а не внутрішній RC-осцилятор (похибка якого може сягати 10%).
04 — Data Frame — структура кадру

Кожен байт передається у вигляді фрейму. Фрейм — це строго визначена послідовність бітів зі службовими маркерами початку і кінця.

UART Data Frame — 8N1 (8 біт, no parity, 1 stop)
START D0–D7 STOP
ЧастинаЗначенняПризначення
Start bit 0 (LOW) Перехід HIGH→LOW будить приймач і запускає його лічильник бодів. Завжди рівно 1 біт.
Data bits 5–9 біт Самі дані, починаючи з молодшого біта (LSB first). Стандарт — 8 біт.
Parity bit опційно Even/Odd — перевірка на непарну кількість одиниць. Якщо не потрібно — вимикається (None).
Stop bit 1 (HIGH) Лінія повертається в IDLE. Якщо приймач бачить Stop=0 — це Frame Error.
05 — Принцип роботи

Лінія UART в стані спокою завжди HIGH (логічна 1). Саме тому перший перехід HIGH→LOW однозначно розпізнається як стартовий біт — ніяка інша частина фрейму не починається з цього переходу.

Oversampling ×16 — як приймач читає біт у центрі ×16 oversampling

Приймач ділить кожен бод на 16 рівних вибірок. Після виявлення стартового біта він чекає 8 вибірок (половину baud'а), щоб синхронізуватись з центром першого біта даних. Далі зчитує кожен наступний біт теж у центрі — через кожні 16 вибірок. Для захисту від коротких шумових імпульсів UART зчитує три вибірки навколо центру (8, 9, 10) і бере більшість — якщо 2 з 3 однакові, це справжнє значення біта.

Покрокова передача байта через модуль UART
  • CPU записує байт у TX буфер (UDR). Якщо зсувний регістр передавання (TSR) вільний — байт одразу туди потрапляє. Якщо зайнятий попередньою передачею — байт чекає.
  • TSR бере байт і починає видавати біти на лінію TX по одному, синхронно до baud rate генератора. Кожен біт тримається рівно 1 baud.
  • Приймач виявляє перехід HIGH→LOW на своєму RX і запускає лічильник вибірок. Чекає ½ бод щоб потрапити в центр.
  • RSR збирає біти з лінії RX, зсуваючи кожен новий біт у регістр. Після 8 бітів байт зібраний.
  • Байт передається з RSR у RX буфер, встановлюється прапорець RXC (Receive Complete). CPU може прочитати байт з UDR.
INF
Якщо CPU не прочитав попередній байт з RX буфера до того, як RSR прийняв наступний — встановлюється прапорець DOR (Data OverRun) і старий байт затирається. Вирішення: переривання на прийом або DMA.
06 — Відстань роботи

RS-232 і RS-485 — це не нові протоколи, а фізичний рівень для UART. Вони додають лінійний драйвер (перетворюють TTL 3.3V/5V на ±12V або диференціальний сигнал) і дозволяють передавати на великі відстані по витій парі.
07 — Будова модуля UART

UART — це окремий апаратний модуль усередині MCU. CPU тільки записує байт у буфер і читає з буфера — все інше (серіалізація, таймінг, старт/стоп-біти) виконується апаратно, паралельно з основним кодом.

КомпонентПризначення
TSR
Transmit Shift Register		 Зсувний регістр передавання. Бере байт з TX Buffer і виштовхує його побітово на лінію TX — один біт за такт baud rate генератора.
RSR
Receive Shift Register		 Зсувний регістр прийому. Збирає біти з лінії RX один за одним. Після отримання повного байта передає його в RX Buffer.
TX Buffer
UDR (write)		 CPU записує сюди байт для відправки. Якщо TSR вільний — байт одразу передається. Якщо зайнятий — очікує. Прапорець UDRE показує чи готовий буфер.
RX Buffer
UDR (read)		 Після прийому байта RSR копіює його сюди. CPU читає звідси. Якщо новий байт прийде раніше ніж CPU прочитає — DOR (Overrun Error).
Baud Rate Generator
UBRR + prescaler		 Лічильник, який відраховує UBRR+1 тактів CPU і генерує "клік". Саме цей клік зсуває наступний біт у TSR або запускає вибірку в RSR.
UCSRxA / B / C UCSRxA — прапорці стану: TXC, RXC, UDRE, FE, DOR. UCSRxB — дозволи переривань, увімкнення TX/RX. UCSRxC — формат кадру (біти даних, parity, стоп-біти).
KEY
Саме тому DMA + UART такий ефективний: CPU записує адресу буфера в DMA один раз і забуває. DMA сам годує байт за байтом у UDR без участі CPU — ідеальний патерн для передачі логів або великих масивів даних.
08 — Переваги та недоліки
Переваги
  • Full-duplex — одночасна передача і прийом по окремих лініях TX і RX.
  • Простота — лише 2 дроти для базового зв'язку, без тактового сигналу.
  • Широка підтримка — є в кожному MCU та більшості модулів (BT, WiFi, GPS, GSM, AT-команди).
  • Довжина лінії — до 15 м на низьких швидкостях без додаткових компонентів (більше ніж SPI/I²C).
Недоліки
  • Тільки 1:1 — не підтримує кілька пристроїв на одній шині (на відміну від I²C чи SPI).
  • Залежність від частоти — без CLK-лінії будь-яка похибка понад 2–3% у baud rate призводить до помилок.
  • Overhead — 2–3 службових біта на кожні 8 біт даних (~20% втрат пропускної здатності).
  • Немає адресації — неможливо підключити кілька пристроїв без мультиплексора або RS-485 шини.
09 — Код
C · STM32 HAL — ініціалізація + передача + прийом у перериванні 
// 1. Ініціалізація (зазвичай генерується CubeMX)
UART_HandleTypeDef huart1;

void MX_USART1_UART_Init(void) {
    huart1.Instance        = USART1;
    huart1.Init.BaudRate   = 115200;
    huart1.Init.WordLength = UART_WORDLENGTH_8B;
    huart1.Init.StopBits   = UART_STOPBITS_1;
    huart1.Init.Parity     = UART_PARITY_NONE;
    huart1.Init.Mode       = UART_MODE_TX_RX;
    huart1.Init.HwFlowCtl  = UART_HWCONTROL_NONE;
    HAL_UART_Init(&huart1);
}

// 2. Передача — блокуючий режим (для налагодження)
uint8_t msg[] = "Hello UART\r\n";
HAL_UART_Transmit(&huart1, msg, sizeof(msg) - 1, 100);

// 3. Прийом у перериванні — правильний підхід для реального проєкту
volatile uint8_t rx_buf;

// Запускаємо прийом 1 байта (викликати один раз при ініціалізації)
HAL_UART_Receive_IT(&huart1, &rx_buf, 1);

// Колбек — автоматично викликається HAL після кожного прийнятого байта
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        ring_buffer_push(rx_buf);              // зберігаємо в кільцевий буфер
        HAL_UART_Receive_IT(&huart1, &rx_buf, 1); // знову чекаємо байт
    }
}

10 — Питання на співбесіді

UART — асинхронний послідовний протокол для зв'язку точка-точка (1:1). Немає тактового сигналу — обидва боки домовляються про baud rate заздалегідь і кожен незалежно відраховує час.

SPI — синхронний, є CLK, MOSI, MISO та CS. Може мати кілька slave пристроїв. Швидший за UART, але потребує більше ліній і не підходить для великих відстаней.

I²C — синхронний, лише 2 лінії (SDA та SCL). Підтримує до 127 пристроїв на шині. Повільніший за SPI, але найефективніший за кількістю пінів для мультипристроєвих систем.

UART найпростіший у налагодженні і найпоширеніший для зв'язку з зовнішніми модулями (GPS, GSM, BT), де немає потреби в шині.

Лінія UART в стані спокою (IDLE) завжди HIGH. Тому перший перехід HIGH→LOW однозначно ідентифікується приймачем як початок передачі — це стартовий біт.

Саме ця домовленість дозволяє приймачу самостійно синхронізуватись на початку кожного байта без зовнішнього CLK. Після виявлення стартового біта він знає коли чекати кожен наступний біт — через рівно 1/baud_rate секунд.

Frame Error (FE) — встановлюється коли стоп-біт прийнятого кадру виявляється LOW замість HIGH. Причина: невідповідність baud rate між передавачем і приймачем (різниця > 2–3%), або шум на лінії під час стоп-біта.

Overrun Error (DOR — Data OverRun) — виникає коли CPU не встиг прочитати попередній байт з RX-буфера до того, як RSR прийняв наступний. Новий байт перезаписує старий — дані губляться. Вирішення: прийом через переривання або DMA, або збільшення кільцевого буфера.

Oversampling ×16 означає, що приймач робить 16 вибірок на кожен бод. Після виявлення стартового біта він чекає 8 вибірок (половину baud'а) щоб потрапити в центр першого біта даних. Далі читає кожен наступний біт теж у центрі — через кожні 16 вибірок.

Чому центр? Краї біта нестабільні — там відбуваються перехідні процеси і невеликі затримки через RC характеристики ліній. Центр максимально далекий від обох країв, тому читання там найнадійніше. Для додаткового захисту від шуму UART читає вибірки 8, 9, 10 і бере majority vote (якщо 2 з 3 однакові — це справжнє значення).

Polling (HAL_UART_Transmit/Receive без _IT) — CPU блокується і чекає поки кожен байт передасться. Підходить тільки для налагодження або сценаріїв де реальний час не критичний.

Interrupt (HAL_UART_Receive_IT) — CPU займається своїм, переривання спрацьовує на кожен прийнятий байт. Підходить для більшості застосунків з помірним трафіком. При великому потоці переривання генеруються дуже часто і навантажують CPU.

DMA — CPU налаштовує DMA один раз і забуває. DMA сам переносить дані між UART і пам'яттю без участі CPU. Ідеально для великих буферів, логування, передачі масивів. Складніша конфігурація, потрібен Circular DMA для безперервного прийому.
